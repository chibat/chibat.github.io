Introducing Doma

August 15, 2017

Doma is a database access framework for Java.

  • github
  • Japanese document

Doma is so cool !
Doma has a Japanese document, but unfortunately there is no English document.
Therefore I introduce Doma in English.

Feature

  • Code generation and SQL file validation at compile time by Pluggable Annotation Processing API
  • SQL template which is called “2-way SQL”.
  • Usable only JRE. Don’t have dependencies.

Doma has many other features.

About 2-way SQL

selectByAgeRange.sql

select /*%expand*/* from employee
where
/*%if min != null */
  age >= /* min */10
/*%end */
/*%if max != null */
  and
  age <= /* max */70
/*%end */
order by age

You can build SQL with SQL comment.
min and max are mapped Java method parameters.

Getting Started

Prerequisites

Install JDK 8.

Clone boilerplate project

Clone simple-boilerplate repository from GitHub.

$ git clone https://github.com/domaframework/simple-boilerplate.git

Dao class

See src/main/java/boilerplate/dao/EmployeeDao.java.

@Dao(config = AppConfig.class)
public interface EmployeeDao {

    @Select
    List<Employee> selectAll();

    @Select
    Employee selectById(Integer id);

    @Insert
    int insert(Employee employee);

    @Update
    int update(Employee employee);

    @Delete
    int delete(Employee employee);
}

You only make Dao interface.
Implemation class is generated by Pluggable Annotation Processing API.

Select

Call Dao method which is annotated @Select, when executing select process.

Check the following code in EmployeeDao.java.

@Select
Employee selectById(Integer id);

Check the following SQL(src/main/resources/META-INF/boilerplate/dao/EmployeeDao/selectById.sql).

select
    /*%expand*/*
from
    employee
where
    id = /* id */0

/*%expand*/* is replaced to field names which is based return type(Entity class) of Dao class.
id is binded to Java method parameters.
0 at the back is the test data.
You can easily check syntax error in SQL with SQL tool, because test data exist.
Test data is not used when executing Java programs.

Check the code in EmployeeDaoTest.java.

@Test
public void testSelectById() {
    TransactionManager tm = AppConfig.singleton().getTransactionManager();
    tm.required(() -> {
        Employee employee = dao.selectById(1);
        assertNotNull(employee);
        assertEquals("ALLEN", employee.name);
        assertEquals(Integer.valueOf(30), employee.age);
        assertEquals(Integer.valueOf(0), employee.version);
    });
}

Execute by JUnit.

$ ./gradlew test

The following SQL is the actual SQL to be executed.

select
    age, id, name, version
from
    employee
where
    id = 1

Insert

Call Dao method which is annotated @Insert, when executing insert process.

Check the following code in EmployeeDao.java.

@Insert
int insert(Employee employee);

Add the following code in EmployeeDaoTest.java

@Test
public void testInsert() {
    TransactionManager tm = AppConfig.singleton().getTransactionManager();

    Employee employee = new Employee();

    // first transaction
    // execute INSERT
    tm.required(() -> {
        employee.name = "HOGE";
        employee.age = 20;
        dao.insert(employee);
        assertNotNull(employee.id);
    });

    // second transaction
    // check that the INSERT was successful.
    tm.required(() -> {
        Employee employee2 = dao.selectById(employee.id);
        assertEquals("HOGE", employee2.name);
        assertEquals(Integer.valueOf(20), employee2.age);
        assertEquals(Integer.valueOf(1), employee2.version);
    });
}

Execute by JUnit.

The following SQL is the actual SQL to be executed.

insert into Employee (age, id, name, version) values (20, 100, 'HOGE', 1)

The identifier and version number are set automatically.

Update

Call Dao method which is annotated @Update, when executing update process.

Check the following code in EmployeeDao.java.

@Update
int update(Employee employee);

Add the following code in EmployeeDaoTest.java

@Test
public void testUpdate() {
    TransactionManager tm = AppConfig.singleton().getTransactionManager();

    // first transaction
    // update the age field after searching
    tm.required(() -> {
        Employee employee = dao.selectById(1);
        assertEquals("ALLEN", employee.name);
        assertEquals(Integer.valueOf(30), employee.age);
        assertEquals(Integer.valueOf(0), employee.version);
        employee.age = 50;
        dao.update(employee);
        assertEquals(Integer.valueOf(1), employee.version);
    });

    // second transaction
    // check that the UPDATE was successful.
    tm.required(() -> {
        Employee employee = dao.selectById(1);
        assertEquals("ALLEN", employee.name);
        assertEquals(Integer.valueOf(50), employee.age);
        assertEquals(Integer.valueOf(1), employee.version);
    });
}

The following SQL is the actual SQL to be executed.

update Employee set age = 50, name = 'ALLEN', version = 0 + 1 where id = 1 and version = 0

The version number for optimistic exclusive control is automatically incremented.

Delete

Call Dao method which is annotated @Delete, when executing delete process.

Check the following code in EmployeeDao.java.

@Delete
int delete(Employee employee);

Add the following code in EmployeeDaoTest.java

@Test
public void testDelete() {
    TransactionManager tm = AppConfig.singleton().getTransactionManager();

    // first transaction
    // execute DELETE
    tm.required(() -> {
        Employee employee = dao.selectById(1);
        dao.delete(employee);
    });

    // second transaction
    // check that the DELETE was successful.
    tm.required(() -> {
        Employee employee = dao.selectById(1);
        assertNull(employee);
    });
}

Execute by JUnit.

The following SQL is the actual SQL to be executed.

delete from Employee where id = 1 and version = 0

In addition to the identifier, the version number is also specified as the search condition.

Eclipse plugin

Doma has a useful Eclipse plugin.
The plugin compile Java sources, when the SQL file is saved.

http://dl.bintray.com/domaframework/eclipse/
